## Abstract

This document describes the architecture of a prediction market recommendation tool.
Large language models, LLMs,
are utilized to create a curated list of relevant prediction markets for each user.
Additionally, the information collected by the recommendation tool
can be used to improve AI-driven insights for each user about a specific market.

## Background/Motivation

With the growth of prediction markets,
users face hurdles in tracking complex portfolios,
identifying relevant events across thousands of active listings,
and understanding volatile market movements.

With artificial intelligence agents, users can improve their prediction market experience.
This recommendation tool allows users to comb through irrelevant markets.
With an intuitive user interface,
prediction market participants can connect their wallets,
analyze their open positions,
discover trending events that they may have missed, and
consult with an AI assistant for real-time market analysis.

## Specification

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in [RFC 2119](http://tools.ietf.org/html/rfc2119).

The user is presented with recommendations that are generated by an AI agent using an LLM model.
The agent SHOULD collect a `profile-schema` and
a `market-schema`.
The `profile-schema` will include important information about the user's preferences and
behavior.
The `market-schema` will include the details about a specific prediction market.

### User Profile

A user's profile SHOULD use a wallet address as the user's identifier, `user_identity`.
The `user_identity` MUST be declared in the user `profile-schema`.
The RECOMMENDED format of the `profile-schema` is as follows:

```js
{
  "user_identity": address,

  // the categories and/or sub-categories
  "explicit_interests": {
    "politics": int,
    "crypto": int,
    "sports": int,
    "macro": int
 },

  // Calculated after each recommendation request
  "inferred_interests": {
    "topic": {
      "interest_score": int,
      "skill_score": int,
      "engagement_score": int
 },
 },

  "market_settings": {
    "binary_markets": int,
    "short_term_markets": int,
    "high_volatility_markets": int,
    "news_driven_markets": int,
    "low_liquidity_markets": int,
    "resolution_type": int
 },

  "performance": {
    "popular_market": [string], // The market category the user interacts with the most
    "wins": int, // number markets resolved in their favor
    "losses": int, // number of markets resolved not in their favor
 },

  "embedding_vector": [0.028, -0.194, 0.338, ...],

  "last_updated": "2026-05-28T18:10:00Z"
}

```

`explicit_interset`:

- it MAY be updated based on the user's activity

`inferred_interest`:

- SHOULD be calculated after each new recommendation list request
- the previous `inferred_interest` MAY be used to update the current `inferred_interest`

`market_settings`:

- SHOULD be set manually by the user (// liquidity scale?)

#### Profile Scoring

Each user will have a unique profile score based on a few factors.
Scoring SHOULD be calculated after the profile owner interacts with the recommendation agent.
Three main areas are being calculated and
re-evaluated for each topic for each profile.

The `interest_score` which calculates the user's assumed interest in a market based on the interactions.
Below is a RECOMMENDED format :

```python

 interest_score =
    0.1 * click_rate +
    0.4 * watchlist_rate +
    0.5 * trade_rate

```

- `click_rate` : The number of clicks a market gets from the user.
- `watchlist_rate`: If the market was bookmarked.
- `trade_rate` : level of exposure if a bet was placed for the market.

The `engagement_score` which calculates how engaged the user is with markets on a list.
Below is a RECOMMENDED format :

```python
  
 engagement_score =
    0.3 * avg_time_spent +
    0.3 * repeat_visits +
    0.2 * watchlist_additions

```

The `skill_score` which calculates the user's performance for markets a user trades in.
Below is a RECOMMENDED format :

```python
  
 skill_score =
      0.4 * calibration
    + 0.3 * roi
    + 0.2 * timing_quality
    + 0.1 * consistency

```

Each profile SHOULD have scores updated periodically.
This will help improve the future recommendation list presented by the LLM.

A user is not REQUIRED to make a mainnet transaction for prediction markets to update profile scoring.
Mainnet unresolved prediction markets SHOULD be presented each time,
but a user may use a local blockchain to record bets that do not have value.
This can be used as practice when a user is not confident about a bet but
still wants to record a prediction.
Since the transaction is local,
the local blockchain MAY not keep transactions if the node shuts down.
For local practice transactions,
it is RECOMMENDED to keep the latest blockchain state stored in a local database.
When the node is started,
it MAY check the local database for the latest state.

#### Prediction Market Connections

The prediction market data REQUIRED for a user to receive recommendations are:

- The prediction market profile
- Current unresolved prediction markets data from prediction markets
- Current open trades on the prediction market, as this can help prevent displaying twice

An example API connection from the Polymarket API documentation is the user's public profile.

### Market Schema

Each prediction market SHOULD be presented as a market schema.
The values SHOULD be derived from a prediction market data source,
like the Polymarket Gamma API mentioned before.
Below is the RECOMMENDED format of the schema:

```js
{
  "id": string,
  "event": string,
  "markets": [
 {
        "question_A": 
 { // the question's description
            "subject": string,
            "condition": string,
            "deadline": int
 }
 }
 ],
  
  "category": string, // main category
  "sub-category": [string],
  "market_type": string, // binary/scalar/sports/multi-outcome
  "time_horizon_days": 1int, // time frame for markets
  "resolution_date": int, // time of event resolution
  "resolution_type": string // event
  
  "embedding": [int],

  "difficulty": int,
  "volatility": int,
  "liquidity": int,

}

```

Matching markets with similar outcomes,
Based on concepts presented in the
[Semantic Trading: Agentic AI for Clustering and Relationship Discovery in Prediction Markets](https://arxiv.org/abs/2512.02436),
the recommendation tool implements a lightweight approach.
The approach is inspired by the paper's market-relation discovery framework.
Instead of implementing the agentic workflow,
we use structured market representations,
embeddings for semantic retrieval,
and an LLM-based relation classifier task to identify markets that are likely to resolve similarly.
This captures the core value of discovering related prediction markets,
while remaining lightweight and easier to deploy.

The system generates pairs of related prediction markets by identifying semantic similarity and
potential outcome equivalence between markets.

```js
{
  "market_i": string,
  "market_j": string,
  "is_same_outcome": true,
  "confidence": int,
  "rationale": string
}

```

The `event`, `markets`, `description`,
and `embedding` values from the market SHOULD be used to generate market pairs.
An `embedding` is a dense,
high-dimensional vector representation of text generated by a neural model.
It maps semantically similar inputs to nearby points in vector space.

Instead of comparing every market pair (O(n²)), this system:

Converts each market into an embedding vector
Computes cosine similarity between vectors
Retrieves top-K nearest neighbors for each market

This reduces candidate pairs from:

> O(n²) → O(n log n) or O(nk)

depending on the retrieval strategy.
Prediction markets often differ in wording but share semantic intent.
For example, a market could describe the price threshold for a trading pair,
while other markets use the “new all-time high” description.
Different descriptions, but both rely on similar resolutions.

If matching occurs after user profile preference matching searches,
some markets may be missed.

- Each match should have a father pair, so it does not end up in the list twice(once with a matched event and another later in the list)

Flow:

- Market is used to create a `description`
- An LLM invoked to extract the `description` fields
- OpenAI SDK is used to create an embedding of the descriptions field
- Embedding is used by the search system to find potential matches
- A second LLM invocation is used to review the potential matches as final matches

For the recommendation list, see the [#recommendations] section for further explanation.
The values used from the market schema:

- id: to organize the final list with identifiers
- market-type: to match the user profile preferred market type
- category and sub-category
- time horizon days
- resolution date
- resolution type
- difficulty
- volatility
- liquidity

### Agent Module

The recommendation tool MAY use transparent agents that share the LLM model being used,
a rationale about the response and
create attestations on-chain.
Users can audit an agent's performance through its on-chain agent identity.
The recommendation agent MUST register through the [ERC-8004](https://eips.ethereum.org/EIPS/eip-8004) Identity Registry.
Each recommendation response SHOULD include the `agentId` used to generate the recommendation,
allowing users to resolve the agent's metadata.
The metadata will include the 'supportTrust` which is the REQUIRED trust model for the recommendation tool.
The only trust model used will be the reputation registry.
The recommendation agent's registration file MUST be formatted as:

```js
{
  "type": "https://eips.ethereum.org/EIPS/eip-8004#registration-v1",
  "name": "Recommendation Agent",
  "description": "Generates prediction market recommendations.",

  "image": string,

  "active": bool,
  "x402Support": bool,

  "registrations": [
 {
      "agentId": int,
      "agentRegistry": address
 }
 ],

  "supportedTrust": [
    "reputation"
 ],

  "capabilities": [
    "recommendations",
 ],
}

```

Each user profile SHOULD be assigned to its own `agentId`.
The ERC8004 agent reputation registry will allow the owner to provide feedback about the agent's responses.

Security Note:
Sybil attacks are possible by parties incentivized to inflate the reputation of an agent.
While an `agentId` is owned by a single owner,
recommendation agents MAY rely on external software dependencies,
models, APIs, and data sources that can be compromised through supply-chain attacks or influenced by data poisoning.
Users may not know if a recommendation response was affected by an attack.
It is RECOMMENDED that users submit feedback to the Reputation Registry when recommendation responses appear suspicious.

#### Agent Response

Each user session is not REQUIRED to generate a new recommendation response.
The previous recommendations response SHOULD be accessible for auditing purposes.
This MAY be on-chain or off-chain.
